## 1. このツールが解決する課題（背景）

生成AI検索（ChatGPT / Perplexity / Google AI Overview 等）が普及し、ユーザーは「検索してリンクをクリックする」のではなく「AIに質問して要約された答えを読む」ようになりました。

この結果、企業にとって新しい死活問題が生まれています：

> **「AIが回答を作るとき、自社のサイトが"出典（引用元）"として使われているか？」**

従来のSEO順位は意味を失いつつあり、代わりに **AEO（Answer Engine Optimization＝AI回答最適化）** という指標が重要になっています。しかし、**「自社がAIにどれだけ引用されているか」を測る手段を、ほとんどの企業は持っていません。**

HackⅡは、この「AI引用度」を測定・可視化するSaaSです。

### 想定顧客と売り方
- **主：** SEO会社・Web広告代理店（自社の提案商材として、複数のクライアントサイトを管理して使う）
- **副：** 事業会社のマーケティング担当者（自社の引用状況を直接モニタリング）
- **収益モデル：** 月額SaaS（プロジェクト数・キーワード数で段階課金）＋ 成果報酬連動

> ⚠️ **設計上の最重要ポイント：** 主要顧客が代理店であるため、**1アカウントで複数のクライアントサイトを管理できる階層構造**を最初から組み込んでください（後付けが最も困難なため）。詳細は §5 のデータモデルを参照。

## 2. システムの基本動作（1枚で理解する）

```
【1】顧客（代理店）が登録する
     - 計測対象サイト（例：client-a.com）
     - 監視したいキーワード群（例：「家族信託 費用」「相続 弁護士 おすすめ」…）
     - 競合サイト（例：competitor-b.com, competitor-c.com）

           ↓

【2】システムが定期的に（週次バッチ）実行する
     各キーワードを AI検索API（主軸：Perplexity）に質問として投げる
     例：質問「家族信託の費用はいくらですか？」

           ↓

【3】AIが「回答」と一緒に「出典URLリスト（citations）」を返す
     citations = [
        "https://competitor-b.com/article/123",
        "https://other-site.com/blog/456",
        "https://client-a.com/family-trust"   ← これがあれば「引用された！」
     ]

           ↓

【4】システムが出典リストを解析（パース）して判定・記録する
     - 自社ドメインが含まれているか？（引用有無）
     - 競合と比べて何%引用されているか？（引用シェア）
     - 先週と比べて増えたか減ったか？（推移）

           ↓

【5】ダッシュボードとCSVで顧客に見せる
```

**つまり「自社サーバーのログ」は一切使いません。** 参照するのは **AI側（Perplexity等）のAPIが返してくる出典データ** です。ここが本ツールの技術的な核心です。

## 3. 取得・計測すべきデータの定義（本書の中核）

「どんなデータが取れれば売れるか」を4階層（L1〜L4）で定義します。

### 🔴 L1：引用有無（MUST）— 提案の入口

| データ項目 | 定義 | 取得方法 |
|---|---|---|
| `is_cited` | あるキーワードのAI回答の出典に、自社ドメインが含まれるか（true/false） | citations配列をドメイン単位で照合 |
| `cited_url` | 自社ドメインのうち、実際に引用された具体的なURL | citations配列から自社ドメイン一致分を抽出 |
| `cited_position` | 出典リスト内での掲載順位（1番目／2番目…） | citations配列のindex |
| `answer_text` | AIが生成した回答本文（証跡として保存） | API応答のテキスト部分 |
| `queried_at` | この計測を実行した日時 | バッチ実行時刻 |

> **売れる理由：** 「御社、このキーワードでAIに全く出てきていません」という事実を1秒で見せられる。営業の入口になる。

### 🔴 L2：引用シェア／SOV（MUST）— 契約の決め手

SOV = Share of Voice（占有率）。同じキーワード群に対して、自社と競合がそれぞれ何%引用されているか。

| データ項目 | 定義 | 計算式 |
|---|---|---|
| `domain_citation_count` | ドメインごとの総引用回数（計測キーワード全体での合計） | 全キーワードのcitationsを集計 |
| `total_citation_slots` | 計測対象の全出典枠数 | Σ（各キーワードのcitations件数） |
| `share_of_voice` | 自社／競合の引用シェア（%） | `domain_citation_count ÷ total_citation_slots × 100` |
| `rank_among_competitors` | 登録ドメイン群の中での引用シェア順位 | シェア降順ソート |

> **売れる理由：** 「競合A社が引用シェア60%、御社は12%」という比較が、最も強い危機感＝発注動機を生む。L1単独より圧倒的に契約率が上がる。

### 🔴 L3：推移トラッキング（MUST）— 継続課金の根拠

| データ項目 | 定義 | 取得方法 |
|---|---|---|
| `snapshot_date` | 計測スナップショットの基準日（週次） | バッチ単位 |
| `weekly_citation_rate` | その週の引用率（is_citedがtrueだったキーワードの割合） | 週次集計 |
| `share_of_voice_trend` | 引用シェアの週次推移（時系列配列） | 各スナップショットのSOVを連結 |
| `delta_vs_previous` | 前週比の増減（ポイント差） | 今週 − 先週 |

> **売れる理由：** 「施策を打ったら引用率が12%→34%に上がった」という変化を見せられて初めて、顧客は来月もお金を払う。**ここがSaaS（継続課金）の生命線。** 単発計測では絶対に解約される。

### ⚪ L4：改善示唆（LATER — 今回スコープ外）

| データ項目 | 定義 |
|---|---|
| `losing_keywords` | 競合が引用されているのに自社が引用されていないキーワード一覧 |
| `gap_pages` | どの競合ページに負けているかの分析 |
| `ai_recommendation` | 「このページをこう直せば引用されやすい」というAI生成の改善案 |

> 今回は作りません。L1〜L3が安定稼働した後の追加発注とします（提案書には「将来構想」として記載いただけると嬉しいです）。

## 4. 対応するAI検索エンジンと取得方針（重要な技術判断）

ここはエンジニアの方に **特に率直なご意見をいただきたい** 部分です。

| AI検索 | citations取得の安定性 | 今回の方針 |
|---|---|---|
| **Perplexity API** | ◎ 出典URLを構造化して安定的に返す | 🔴 **MVPの主軸。まずここを確実に動かす** |
| **Google AI（Gemini / AI Overview）** | △ 標準APIでは出典が安定的に取れない場合あり | 🟡 取得可能な範囲で。困難なら「対応予定」表記で可 |
| **OpenAI（GPT / ChatGPT検索）** | △ Web検索連携モードで部分的に取得可 | 🟡 同上。コストと安定性次第 |

**発注者としての割り切り：**
「全AIに完璧対応」は今回求めません。**Perplexityで確実に動く計測ツールが1本完成すること**を最優先とし、GPT/Geminiは「取れる範囲で」「段階的に」で構いません。
→ もし「この方針だと予算内で現実的」「いや、ここは難しい」というご意見があれば、提案段階で必ず教えてください。

## 5. データモデル（DB設計の骨子）

代理店が複数クライアントを管理する前提を、**最初から**構造に入れてください。

```
account（アカウント＝代理店 or 事業会社）
   └── project（プロジェクト＝計測対象の1サイト／クライアント）
          ├── target_domain（自社ドメイン）
          ├── competitor_domains（競合ドメイン：複数）
          ├── keyword（監視キーワード：複数）
          └── snapshot（週次計測スナップショット：複数）
                 └── citation_result（キーワード×実行ごとの計測結果）
```

### 主要テーブル定義（参考イメージ・最終設計はお任せします）

**accounts（アカウント）**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| name | VARCHAR | 代理店名／会社名 |
| plan | VARCHAR | 契約プラン（free / standard / agency 等） |
| created_at | TIMESTAMP | 作成日時 |

**projects（プロジェクト＝1クライアントサイト）**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| account_id | UUID | accountsへの外部キー |
| name | VARCHAR | プロジェクト名（例：A社サイト計測） |
| target_domain | VARCHAR | 計測対象の自社ドメイン |
| created_at | TIMESTAMP | 作成日時 |

**competitors（競合ドメイン）**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| project_id | UUID | projectsへの外部キー |
| domain | VARCHAR | 競合ドメイン |

**keywords（監視キーワード）**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| project_id | UUID | projectsへの外部キー |
| keyword | VARCHAR | 計測キーワード |
| query_template | TEXT | AIに投げる質問文（例：「{keyword}について教えて」） |
| is_active | BOOLEAN | 計測ON/OFF |

**citation_results（計測結果：本ツールの心臓部）**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| keyword_id | UUID | keywordsへの外部キー |
| ai_engine | VARCHAR | 計測したAI（perplexity / gpt / gemini） |
| snapshot_date | DATE | 計測基準日（週次） |
| queried_at | TIMESTAMP | 実際の実行時刻 |
| answer_text | TEXT | AIの回答本文（証跡） |
| citations_raw | JSONB | 返ってきた出典URL配列（生データを丸ごと保存） |
| is_cited | BOOLEAN | 自社ドメインが出典に含まれたか |
| cited_url | VARCHAR | 引用された自社URL |
| cited_position | INTEGER | 出典内での順位 |

> 💡 **citations_raw を JSONB で丸ごと保存しておくこと**を強く希望します。後から「競合の何が引用されたか」を再集計したくなった時、生データがあれば再計算できます。データは捨てずに貯める方針でお願いします。

## 7. 計測の頻度・運用

| 項目 | 要望 |
|---|---|
| 計測頻度 | 週次バッチ（毎週決まった曜日に自動実行） |
| 手動計測 | 任意のタイミングで「今すぐ計測」できるボタン（🟡 SHOULD） |
| 1プロジェクトあたりキーワード数上限 | 50〜100件程度を想定（プラン次第・設計はお任せ） |
| API利用コスト管理 | Perplexity等のAPI課金が青天井にならないよう、計測回数の上限／予算アラートを設けたい（🟡 SHOULD） |

> ⚠️ **API利用料は運用コストに直結します。** 「キーワード100件 × 顧客50社 × 週4回」でAPIを何回叩き、月いくらかかるのか、提案段階で概算を出していただけると非常に助かります。
